# Decision Record

Design choices for this project and the reasoning behind them. The theme is
deliberate trade-offs: this is a portfolio build tuned for cost and clarity,
with the production alternative noted for each choice so the reasoning is
visible rather than hidden.

## Why ECS Fargate (not EKS or plain EC2)

Fargate runs containers without managing servers or a Kubernetes control
plane. For a single small service, EKS is overkill (control-plane cost plus
operational overhead) and plain EC2 means patching and scaling instances by
hand. Fargate is the least-effort way to run a container in a private subnet
behind an ALB, which is exactly the shape of this project.

**Production alternative:** at meaningful scale or with many services, EKS
becomes worth the overhead. For one service, Fargate wins.

## Networking: 2 AZs, single NAT gateway

The VPC spans two availability zones with a public and private subnet in each.
Tasks and the database run in the private subnets; only the ALB and the NAT
gateway sit in public subnets.

**Single NAT trade-off:** one NAT gateway (~$32/month plus data) is a single
point of failure — if its AZ fails, private subnets lose outbound internet.
One NAT per AZ removes that risk but roughly doubles the fixed NAT cost. For a
demo, one NAT is the sensible economy; the code comments and this record flag
it so the awareness is explicit. Moving to per-AZ NAT is a small change (a NAT
and route table per AZ).

## Security groups: three tiers, chained

Three security groups, each only allowing what the tier above it needs:

- **ALB SG** — allows inbound HTTP from the internet on 80.
- **ECS SG** — allows inbound only from the ALB SG on the container port.
- **RDS SG** — allows inbound only from the ECS SG on 5432.

Nothing allows blanket access. The database cannot be reached from the
internet at all — only the ECS tasks can talk to it, and only the ALB can
talk to the tasks. This is the standard defence-in-depth layering.

## Database: RDS PostgreSQL, single-AZ, encrypted, private

RDS rather than a self-managed database on EC2: managed backups, patching, and
failover are worth more than the small cost premium.

- **Single-AZ** keeps cost near $15/month. Multi-AZ (a `db_multi_az` toggle)
  adds automatic failover but roughly doubles cost — the production choice,
  off by default here.
- **Encrypted at rest** with a customer-managed KMS key (rotation enabled).
- **Not publicly accessible**; lives in the private subnets.
- **Credentials generated by Terraform** (`random_password`) and stored in
  Secrets Manager — never hardcoded, never committed. The ECS task reads them
  at runtime via the execution role, which can read only this one secret.

## Secrets: Secrets Manager injected into the task

The task definition pulls `DB_HOST`, `DB_USER`, and `DB_PASSWORD` from the
Secrets Manager secret as individual container secrets. The app reads them
from environment variables and never sees a hardcoded credential. Locally, the
same variables come from a `.env` file (git-ignored) — so the app code is
identical in both environments.

## IAM: split execution and task roles, least privilege

- **Execution role** — used by the ECS agent to pull the image and ship logs.
  It can read only this app's DB secret and decrypt with only the RDS KMS key.
- **Task role** — the app's own runtime identity. The app talks to RDS over
  the network, not via AWS APIs, so this role is intentionally near-empty.

Splitting them means the running app has almost no AWS permissions even though
the platform around it needs some.

## Observability: logs plus three alarms

Container logs ship to CloudWatch via the awslogs driver; VPC flow logs
capture network traffic. Three alarms cover the obvious failure modes: ECS CPU
high, ALB target 5xx errors, and RDS CPU high. Alarms publish to an SNS topic;
wiring an email or Slack subscription is a one-line addition left to the
operator.

## Cost-audit tool: pure logic + thin Boto3 layer

The auditor is split so the decision logic (`checks.py`) has no AWS calls and
is fully unit-tested, while a thin Boto3 layer (`auditor.py`) fetches live
data and feeds it in. This is what makes the logic testable without AWS.

It is **read-only by design** — it identifies unattached volumes, idle Elastic
IPs, NAT gateway cost, untagged resources, and RDS sizing, and prints an
estimated monthly figure for prioritisation. It never deletes or modifies
anything; remediation stays a human decision. The dollar figures are rough
estimates for ranking findings, not billing numbers, and the code says so.

## The application is deliberately trivial

The API is just `/health`, `GET /items`, and `POST /items` against Postgres.
That is on purpose: the point of this project is the infrastructure —
networking, security layering, secrets, containers, deployment — not
application features. A more elaborate app would add noise without
demonstrating anything more about the platform.
